# Roadmap

The ordered backlog for the north star (`docs/NORTH-STAR.md`), executed continuously per the autonomy contract (`docs/AUTONOMY.md`).

Item format: `### <id> <title>` followed by `status`, `priority` (P0 highest), `acceptance`. Agents take the topmost `ready` item they can verify. Humans reorder; agents do not change priorities, only statuses and new items at the bottom of the matching milestone.

### M3-6 Durable per-society audit storage (with isolation)
- status: done (2026-07-12; audit-sink API on apps/landing, not per-society Marketplace storage -- see the rejected alternative below)
- priority: P1
- acceptance: a provisioned society's local audit log survives serverless instance recycling without sharing credentials that can read other tenants' data. Today the starter falls back to in-memory when `KV_REST_API_URL`/`KV_REST_API_TOKEN` are absent, so in production the agent's entries vanish between invocations and the cockpit's "Acciones recientes" reads empty (found live 2026-07-09 running the M3-4 proof). The KV key is already namespaced by `SOCIETY_ID`, but handing every society the SAME KV credentials would let any society read the whole store, studio's account records included: that is acceptable ONLY for the platform-owned dogfood society. The real fix provisions isolated storage per society (own KV via the Vercel Marketplace at deploy time, or an authenticated per-society audit-sink API) and wires it in provisioning.
- note: rejected the Marketplace-storage-at-deploy-time path first. Vercel's programmatic Marketplace provisioning (`/docs/integrations/create-integration/marketplace-api`) is a two-sided protocol: Vercel calls the STORAGE PROVIDER's own backend (Upstash's), and the provider calls Vercel back with OIDC/installation credentials issued at partner onboarding. `VERCEL_PROVISION_TOKEN` (a plain personal/team API token, see `apps/studio/src/lib/vercel-provision.ts`) is not that credential and never will be without ar-agents becoming a registered Vercel integration partner -- out of scope. `vercel install <integration>` (the CLI path) is interactive/dashboard-oriented, not scriptable from a server route either. So per-society Marketplace Redis is not feasible with the tokens this project has; if it ever becomes attractive, an integration-partner relationship with Vercel/Upstash is the prerequisite, tracked here as the upgrade path, not attempted.
- note: shipped instead -- an audit-sink API on apps/landing (`POST /api/society-audit/append`, `GET /api/society-audit/tail`, `src/lib/society-audit.ts`), authenticated by the society's OWN existing gate token (`SOCIETY_GATE_TOKEN`/`@/lib/gate-token`, already minted at constitution and already sent to `/api/approvals/gate`), fail-closed with no legacy carve-out (this is a new endpoint; every society already has a gate token). Storage is namespaced `society-audit:{societyId}` in landing's own KV, capped server-side (200 entries, per-field length caps -- never trusts the writer even though the starter already redacts client-side) independent of what the caller claims. The starter's `audit-middleware.ts` (`withLocalAudit`) now dual-writes every tool-call entry: local KV/in-memory (unchanged) plus a best-effort POST to the sink, tracked by a separate `sinkAuditDroppedWrites` counter so a failure on either leg is visible, never silent, and never breaks the tool call itself (both writes are try/catch-wrapped, sink call has a 4s timeout). `GET /api/status`'s audit section reads the local log first (no network cost when non-empty) and falls back to the sink's tail only when local reads empty, surfaced as a new `sinkDroppedWrites` field alongside the existing `droppedWrites`. `appendLocalAudit` now always returns the constructed/signed entry (never null on a local storage failure) so the exact same id/ts/hmac forwards to the sink regardless of local outcome. Isolation: a society can only write/read the namespace its own gate token unlocks -- knowing another society's id (public, appears in audit links) is not enough, proven by cross-token 403 tests. Tests: apps/landing gained 22 (society-audit.ts lib + both routes: auth fail-closed, cross-token isolation, caps, bad JSON); apps/sociedad-ia-starter gained 16 net new (audit-sink.test.ts: 9 new; audit-middleware.test.ts: +4 dual-write/failure-isolation; status.test.ts: +3 sink-fallback, existing exact-shape assertions updated for the new field). Full suites: landing 604/604, starter 69/69. typecheck/test/build green for both apps; apps/studio untouched (its cockpit reads `/api/status`, whose shape only gained a field) and reverified with typecheck.
