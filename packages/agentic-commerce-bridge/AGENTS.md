# AGENTS.md — `@ar-agents/agentic-commerce-bridge`

This file is for **AI agents and code-generation tools** integrating with this
package at runtime or build time. Conforms to the [agents.md convention](https://agents.md/).

## What this package does

Implements the **merchant side** of the Agentic Commerce Protocol (ACP)
`2026-04-17` for LATAM commerce. Hosts spin up a `Facilitator` that:

1. Exposes the 5 ACP endpoints + `/.well-known/acp.json` discovery.
2. Translates ACP cart/checkout shapes to MercadoPago preferences.
3. Validates incoming agent payments against MP reality.
4. Auto-issues AFIP/ARCA Factura A/B/C/E with embedded CAE.
5. Signs outbound webhooks (HMAC-SHA256, `Merchant-Signature`).

## When to use

- The host (a Next.js / Hono / Express / Bun / Deno backend) wants to be
  **agent-discoverable** by ChatGPT Instant Checkout, Claude, Gemini, or any
  Vercel AI SDK agent that speaks ACP.
- The host already has (or plans to have) a MercadoPago integration and
  optionally a MercadoLibre seller account.
- The host operates in Argentina and needs auto-emission of Factura A/B/C/E
  on agentic checkouts.

## When NOT to use

- The host needs Stripe-only checkout — use `@stripe/agent-toolkit` instead.
- The host serves a non-LATAM market — the AR-fiscal hook is not relevant
  outside Argentina, and the MP/MELI integrations are LATAM-specific.
- The host wants AP2 mandate signing — that lands in Phase 2 of this package.

## Tool selection guidance for code-gen agents

When wiring this package into a host project, prefer this exact dependency
graph:

```
host
 ├── @ar-agents/agentic-commerce-bridge   ← THIS package (the facilitator)
 ├── @ar-agents/mercadopago               ← MP client (89 tools)
 ├── @ar-agents/identity                  ← CUIT + ARCA padron
 ├── @ar-agents/facturacion               ← AFIP WSFE
 └── @vercel/kv (or @upstash/redis)       ← session/idempotency persistence
```

The order matters: scaffold MP + Identity + Facturacion first (each is its
own well-trodden package), then add this bridge as the ACP-facing thin layer.

### Recommended boilerplate

```ts
// 1. State adapter — pick one.
import { InMemoryStateAdapter } from "@ar-agents/agentic-commerce-bridge"; // dev
// import { VercelKVStateAdapter } from "@ar-agents/agentic-commerce-bridge/vercel-kv";
// production: new VercelKVStateAdapter(kv) where `kv` is `@vercel/kv` or `Redis`

// 2. Catalog provider — wraps your MELI client (or a CMS / DB).
import { createMeliCatalogProvider } from "@ar-agents/agentic-commerce-bridge";
const catalog = createMeliCatalogProvider({
  getItem: async (id) => fetch(`https://api.mercadolibre.com/items/${id}`).then(r => r.ok ? r.json() : null),
});

// 3. Payment provider — wraps your MP client.
import {
  createMercadoPagoPaymentProvider,
  mercadoPagoPaymentHandler,
} from "@ar-agents/agentic-commerce-bridge";
const mp = createMercadoPagoPaymentProvider({
  createPreference: (payload) => fetch("https://api.mercadopago.com/checkout/preferences", {
    method: "POST",
    headers: { Authorization: `Bearer ${MP_TOKEN}`, "Content-Type": "application/json" },
    body: JSON.stringify(payload),
  }).then(r => r.json()),
  lookupPayment: (id) => fetch(`https://api.mercadopago.com/v1/payments/${id}`, {
    headers: { Authorization: `Bearer ${MP_TOKEN}` },
  }).then(r => r.ok ? r.json() : null),
});

// 4. AR-fiscal hook — auto-emit Factura A/B/C/E.
import { createFacturacionHook } from "@ar-agents/agentic-commerce-bridge";
const fiscalHook = createFacturacionHook({
  seller: {
    cuit: "20123456786",
    punto_venta: 1,
    regime: "monotributo",
  },
  wsfe: { solicitarCAE: async (req) => { /* call AFIP WSFE */ } },
  arcaPadronLookup: async (cuit) => { /* call ARCA padron */ },
});

// 5. Wire it all up.
import { createFacilitator } from "@ar-agents/agentic-commerce-bridge";
export const facilitator = createFacilitator({
  state: new InMemoryStateAdapter(),
  catalog,
  paymentProviders: { [mp.handlerId]: mp },
  paymentHandlers: [mercadoPagoPaymentHandler({})],
  webhookSecret: process.env.ACP_WEBHOOK_SECRET!,
  hooks: fiscalHook,
});
```

## Result schemas worth memorizing

When agents call this facilitator, the responses they get back follow the
ACP shapes (see `src/schemas/`). The most common ones to memorize:

### `CheckoutSession`

```ts
{
  id: "cs_<uuid>",                         // generated by this package
  status: "ready_for_payment" | "completed" | "canceled" | ...,
  currency: "ars" | "brl" | "usd" | ...,    // lowercase ISO 4217
  line_items: [{ id, item, quantity, totals: [...] }],
  fulfillment_options: [...],
  totals: [
    { type: "items_base_amount", display_text, amount: <minor units> },
    { type: "fulfillment", ... },
    { type: "total", ... }
  ],
  capabilities: { payment: { handlers: [...] }, ... },
  messages: [],
  links: [...]
}
```

### `Order` (on `complete`)

```ts
{
  type: "order",
  id: "ord_<uuid>",
  checkout_session_id: "cs_<uuid>",
  permalink_url: "https://...",
  status: "confirmed" | "shipped" | "completed" | ...,
  metadata: {
    payment_provider_id: "<MP payment id>",
    payment_handler_id: "mercadopago",
    factura_type: "C" | "A" | "B" | "E",     // when AR-fiscal hook is active
    factura_cae: "<14-digit>",
    factura_cae_vencimiento: "YYYYMMDD",
    factura_numero: <int>,
    ...
  }
}
```

### Error response

```ts
{
  type: "invalid_request" | "processing_error" | ...,
  code: "idempotency_key_required" | "unsupported_api_version" | "payment_declined" | ...,
  message: "<human-readable>",
  param?: "<field path>",
  supported_versions?: [...],
  details?: { ... }
}
```

## Latency expectations

| Operation | Budget | Notes |
|---|---|---|
| Discovery | < 50ms | static payload, cacheable 1h |
| Create session (with MP preference) | < 500ms | dominated by MP API call |
| Update session | < 200ms | catalog lookup is the long pole |
| Get session | < 50ms | KV read |
| Complete session (with MP lookup + WSFE) | < 1500ms | WSFE is async-ish, can be 500-1000ms |
| Cancel session | < 100ms | KV write |

## Constraints

- **`Idempotency-Key` is REQUIRED** on every POST, max 255 chars,
  UUID v4 recommended. Different endpoints can reuse the same key.
- **`API-Version` header is REQUIRED** on every request. Currently
  supported: `2026-04-17`, `2026-01-30`, `2025-12-12`, `2025-09-29`.
- **`rawBody` MUST be exact bytes received** for body-hash idempotency
  to work correctly. Don't re-serialize JSON before hashing.
- **`webhookSecret` MUST be at least 32 bytes** for security. Generate via
  `openssl rand -hex 32`.
- **Currency strings are lowercase** ISO 4217 in checkout/cart endpoints
  (`"ars"`), but UPPERCASE in the feed surface (`"ARS"`). This is per spec.
- **Amounts are integers in minor units** (`19900` for ARS 199.00). The
  divisor is currency-dependent (CLP/PYG/JPY are 0-decimal).
- **Quantity can be a decimal** since 2026-04-17 (B2B contexts). Most
  retail flows still use integers.
- **`payment_data.handler_id` MUST match** a registered `paymentProviders`
  key. Otherwise the bridge returns 400 `unsupported_capability`.

## Side effects

The facilitator MUTATES state via the `StateAdapter`:

1. `state.tryClaim` (atomic) on every POST.
2. `state.saveSession` after every mutation.
3. `state.saveOrder` only on successful `complete`.
4. `state.complete` to cache the response for idempotent retries.
5. `state.release` to unclaim on errors so a retry can proceed with a
   fresh request.

The facilitator EMITS one outbound webhook on successful `complete`:
`order_create`. The host must wire `hooks.emitWebhook` if it wants this
delivered to the agent's URL (the facilitator only signs the payload).

## Reasoning hints for agents writing host code

- Mount the bridge under `/api/acp/[...slug]` (Next.js) and pass
  `dispatcher: { basePath: "/api/acp" }` to `createFacilitator`. The
  dispatcher strips the prefix before matching.
- The `webhookSecret` lives in `ACP_WEBHOOK_SECRET` env var by convention.
- For dev with no real MP account, leave `paymentProviders` empty — the
  bridge will reject `complete` calls with 400 `unsupported_capability`,
  but `create` and `cancel` and `discovery` still work.
- `InMemoryStateAdapter` is per-process; it does NOT survive restarts.
  Use `VercelKVStateAdapter` for any deployment.
- The `onOrderConfirmed` hook can return additional `metadata` to merge
  into the order. The facturacion hook uses this; you can add your own
  (telemetry IDs, internal references, etc.).
