## 1 · The gap RFC-006 fills

RFC-004 §4 specifies an append-only, per-entry HMAC-signed operational log and,
in §11, explicitly leaves open whether stores allowing random-access overwrite
need a Merkle-chain to give the same guarantee, plus aggregation /
anti-equivocation. A production sociedad-IA wants strictly more than per-entry
tamper-evidence:

- **Insertion / deletion / reordering detection**, not just per-record mutation
  detection.
- **Defensibility when the adversary is the operator itself.** A per-entry (or
  per-link) HMAC is computed with the operator's key; the key-holder can
  re-sign a rewritten history. **The chain alone does not solve this** — see
  §4.0. Only the external anchor (§6) does, and only relative to the last
  notarized checkpoint. RFC-006 states this honestly rather than claiming more.

RFC-006 delivers interior integrity via a **linked HMAC hash-chain** (§3–4) and
operator-defense via a **MUST external anchor sub-chain** (§6). It is a strict
superset of RFC-004: every RFC-006 ledger projects, by the normative
deterministic map in §5, onto RFC-004-conformant entries, so a regulator
holding only RFC-004 and the published RFC-004 tooling can verify an RFC-006
producer with no new software.

## 2 · Canonical-JSON (normative — RFC-006 tightens RFC-004 §3)

A standard whose signature predicate is runtime-dependent cannot anchor
liability law. RFC-006 therefore **tightens** RFC-004 §3 into a form two
independent implementations cannot disagree on:

1. **Code-point key ordering (MUST).** Object keys are ordered by Unicode
   **code point**, not UTF-16 code unit. (JavaScript's default `Array.sort()`
   is code-unit and disagrees with Python/Go/RFC-8785 on astral-plane keys —
   a silent cross-implementation divergence that would produce false
   "tampered".)
2. **Restricted signable domain (MUST).** The only canonicalizable values are:
   `string`, `boolean`, `null`, finite **safe-integer** `number`
   (`Number.isSafeInteger`), `array` thereof, and plain `object` thereof.
   Floating-point, non-finite, `> 2^53`, `undefined`, functions, and symbols
   are **out of domain**. `canonical()` **MUST throw** on an out-of-domain
   value — it MUST NEVER emit non-JSON (`{"a":[1,,2]}`, a bare `undefined`
   token, `NaN`→`null`).
3. **String form.** Minimal JSON escaping, UTF-8, no `\uXXXX` except the
   JSON-mandatory control characters (the RFC-8785 string rule).

The reference producer's `JSON.stringify(sort(v))` is a conformant
implementation **only within this domain and with code-point ordering**;
`arg-verify` reimplements this form clean-room and the generator mirrors it, so
generator and verifier cannot drift. Full RFC-8785 (JCS) number/​string
canonicalization is the v1.1 cross-language target; v1 achieves runtime
independence by *restricting the domain* instead.

## 3 · Chain link (normative)

The unit of an RFC-006 ledger is a **link**, not a free-standing entry.

```
interface ChainLinkPayload {
  seq:        number;            // 1-based, contiguous, no gaps
  prevHash:   string;            // hash of link seq-1; "GENESIS" for seq 1
  societyId:  string | null;     // tenant; null = global ledger (§5 reserves
                                 //   "GLOBAL-LEDGER" and "soc-"-prefixed)
  actor:      string;            // who/what initiated
  action:     string;            // the operation that produced the effect
  meta:       unknown;           // §2-domain; SHOULD carry meta.governance (§5)
  ts:         string;            // ISO-8601 UTC, server clock at write time
}
```

The link authenticator is computed over the **object-canonical** form (no
string concatenation):

```
hash_n = HMAC_SHA256( AUDIT_SECRET,
           canonical({ seq, prevHash, societyId, actor, action,
                       meta: meta ?? null, ts }) )            // lowercase hex
prevHash_1 = "GENESIS"
prevHash_n = hash_{n-1}        (n > 1)
```

`hash_n` is raw lowercase hex (no `sha256:` prefix — that is RFC-004's
per-entry convention). `meta` is normalized to `null` when absent *before*
canonicalization. The only mutation primitive is `append`; no `update`, no
`delete` (TTL-purge of whole prefixes for RFC-004 §7 retention is the sole
exception, never interior links).

## 4 · Chain verification (normative)

**4.0 What the chain does and does NOT prove (read first).** A passing
contiguous chain proves the records are **unmutated, internally linked, and
ordered**. It does **not**, by itself, defend against the key-holding
operator: that operator can **tail-truncate** (drop the most recent links) or
**wholesale-rewrite** from genesis, and a bare verifier will return
`valid:true` for the resulting clean prefix / fresh history. Completeness
against the operator is provable **only** via a verified external anchor
(§6) — see `verifyChainAnchored`. RFC-006 makes this limitation explicit;
implementations MUST NOT advertise operator-defense from the chain alone.

**4.1 Contiguous chain (interior integrity).** A verifier MUST reject a
non-array or **empty** input, MUST require `links[0].seq === 1` and
`links[0].prevHash === "GENESIS"` (rejecting a truncated head / non-rooted
slice), and for every `i` MUST assert: (1) `seq_i == seq_{i-1}+1`
(contiguity); (2) `prevHash_i == hash_{i-1}` (linkage); (3)
`hash_i == HMAC(secret, canonical(payload_i))` (authenticity). It returns
`{valid:true, count, headSeq, headHash}` or `{valid:false, brokenAtSeq, reason}`.

**4.2 Anchored verification (operator-defense).** `verifyChainAnchored(links,
secret, anchors)`: §4.1 MUST pass, the anchor chain (§6) MUST verify, and the
chain head `(headSeq, headHash)` MUST equal the head covered by the **latest
verified anchor**. With no anchors the result is `valid:false` with reason
"completeness/operator-defense not provable" — never a misleading pass.

**4.3 Per-record (non-contiguous slice).** For a filtered view (one society
pulled from a global chain) contiguity cannot hold; a verifier asserts only
check (3) per record and MUST label the result `recordsOnly: true`. This
proves per-record authenticity **and explicitly NOT** set-completeness; it
MUST NOT be presented as a completeness proof.

## 5 · RFC-004 projection (normative — the conformance bridge)

An RFC-006 producer is RFC-004-conformant **by construction** via this
deterministic projection `P`. Given chain link `L`:

```
id         := `${L.ts}-${L.hash.slice(0,16)}`        // 64-bit; see Injectivity
societyId  := MUST be string | null. Any other type → REJECT (no String()
                coercion: `1` and `"1"` must not collide). The values
                "GLOBAL-LEDGER" and any "soc-"-prefixed string are RESERVED
                and forbidden as a tenant societyId (they alias a derived id).
sessionId  := string matching /^[A-Za-z0-9_-]{8,64}$/  → that string
              null                                     → "GLOBAL-LEDGER"
              other valid string                       → "soc-" +
                                base64url(sha256(societyId)).slice(0,16)
ts         := L.ts
tool       := L.action
governance := L.meta.governance if it is an RFC-004 §6 value;
              else "requires-confirmation"  (see Liability-safe below)
input      := { actor: L.actor, seq: L.seq, meta: L.meta ?? null }
              (+ governanceInferred:true when governance was defaulted)
output     := (omitted)
hmac       := "sha256:" + HMAC_SHA256( PROJECTION_SECRET,
                            canonical(stripForSign(P(L))) )
```

Properties (machine-checked by `arg-verify project` / `vectors`):

- **Determinism.** `P` is a pure function of `L`.
- **RFC-004 validity.** `P(L)` is a well-formed `OperationalLogEntry`;
  `verifyEntry(P(L), PROJECTION_SECRET)` per RFC-004 §3 returns `true`. The
  verifier checks **its own** `P(L)` output (not a supplied one) so the
  bridge is proven, not assumed.
- **Injectivity.** `P` is injective over distinct `(societyId|null, seq)`:
  `input.seq` disambiguates within a tenant and `id` embeds 64 bits of
  `hash`. The earlier 32-bit `id` had a birthday collision at ~77 k
  same-`ts` links and is fixed here. (`id` uniqueness is collision-resistant,
  not a hard injectivity proof on its own; `input.seq` is the load-bearing
  disambiguator.)
- **Liability-safe default (MUST).** A producer **MUST** carry
  `meta.governance` (RFC-004 §6 enum). When absent, `P` defaults to
  `requires-confirmation` — the **most operator-onerous** class — and sets
  `input.governanceInferred = true` so the inference is auditable. RFC-006
  does **not** default to `audit-logged`: per RFC-004 §6 that class *shares*
  liability with the LLM provider and would *under-state* operator exposure
  for a real human-confirmed action. The default never moves toward
  `mocked-upstream` or a liability-sharing class.

`PROJECTION_SECRET` MAY equal `AUDIT_SECRET` or be a separate RFC-004 signing
key; the projection is a *view*, the native chain remains the stronger
guarantee. `arg-verify entry P(L).json --secret PROJECTION_SECRET` gives a
regulator a green check with zero RFC-006 awareness.

## 6 · External anchoring (normative — MUST for operator-defense)

The producer **MUST** periodically checkpoint the chain head into an
**anchor**; anchors form their own HMAC-signed chain:

```
interface AnchorBody {
  seq:        number;      // 1-based, contiguous
  headSeq:    number;      // chain head seq at checkpoint time
  headHash:   string;      // chain head hash at checkpoint time
  prevAnchor: string;      // previous anchor signature; "GENESIS" for seq 1
  ts:         string;      // ISO-8601 UTC
}
signature_n  = HMAC_SHA256( AUDIT_SECRET, canonical(AnchorBody_n) )  // hex
prevAnchor_n = signature_{n-1}
```

Anchor verification mirrors §4.1 (reject empty / non-`seq:1` /
non-`GENESIS` / non-contiguous; recompute every signature).

**Minimum external-notary properties (MUST, was an open question).** The
anchor chain MUST additionally be mirrored to an **external notary** that is:
(a) **append-only** with an independently checkable proof of non-removal;
(b) **outside the operator's unilateral control** (a distinct legal/technical
party, a public timestamping authority, or a public transparency log);
(c) **independently fetchable** by a regulator without the operator's
cooperation. Operator-defense (§4.2) holds **only** up to the most recent
head a compliant notary attests. Anything newer than the last notarized
anchor is, by construction, within the operator's tamper window and MUST be
treated as provisional.

## 7 · Asymmetric attestation (normative, aligns RFC-005)

When an RFC-006 producer publishes a signed compliance attestation over the
chain head, it **MUST** use RFC-005 wire conventions: Ed25519 value
base64url-unpadded; public key set at `/.well-known/sociedad-ia/keys`
(RFC-005 §4 JSON: `keyId`, `alg`, `publicKey` SPKI base64url, `validFrom`,
`validUntil`); each signature carries a resolvable `keyId` enabling rotation.
The reference producer currently emits base64 + an embedded key at
`/api/audit/pubkey`; that is a tracked, pure encoding/endpoint divergence
(CONFORMANCE.md), not a design difference.

## 8 · Verification interface (extends RFC-004 §5)

In addition to RFC-004 §5: `GET /api/audit/verify` (§4.1 result);
`GET /api/audit/anchor` (§6 anchor chain, for §4.2); `GET
/api/audit/{slug}/attestation` (§7); and a **projection export** emitting
`P(L)` so a regulator runs RFC-004 tooling unchanged. v1.1 normative-izes the
exact route + content-type; v1 requires the export exist and be documented.

## 9 · Conformance test vectors

`/test-vectors/rfc-006-v1.json` (published, byte-exact, regenerable via
`tools/arg-verify/_gen-rfc006-vectors.mjs`) contains, and `arg-verify
vectors` independently reproduces, all PASS:

1. **Chain.** Genesis-rooted 3-link chain (expected `hash`/link);
   `verifyChain` `valid:true`. **Mutated** copy → `valid:false` at the
   mutated `seq` (hash mismatch). **Deleted-interior** copy → `valid:false`
   (contiguity/linkage).
2. **Anchored.** `verifyChainAnchored(chain, anchors)` `valid:true` (head ==
   latest verified anchor). **Truncated** copy (tail link dropped): bare
   `verifyChain` `valid:true` (a clean prefix — proving §4.0) **and**
   `verifyChainAnchored` `valid:false` (head ≠ anchor) — the operator-defense
   demonstrated end to end.
3. **Records-only.** A non-contiguous per-society slice: every record
   authentic, `verifyChain` `valid:false` — the §4.3 non-guarantee made
   explicit so it cannot be mistaken for completeness.
4. **Anchors.** 2-anchor chain (expected `signature`/anchor); tamper breaks
   verification; empty/non-rooted rejected.
5. **Projection.** For each link the exact `P(L)` and its RFC-004 `hmac`; the
   verifier reproduces `P(L)` itself and that output passes RFC-004 §3
   `verifyEntry`.

## 10 · Decision request

- RFC-004 v1 remains the **minimum** floor.
- RFC-006 v1 is the **recommended profile** for any sociedad-IA handling real
  value, regulated activity, or third-party funds. An RFC-006 producer is
  RFC-004-conformant via §5, so citing RFC-006 does **not** fork the
  standard.
- Suggested language: *"the operational log MUST satisfy RFC-004 v1;
  producers handling [regulated thresholds] MUST satisfy RFC-006 v1
  (including its §6 external-notary requirement)."*

## 11 · Open questions

- **Projection route (v1.1).** Normative path + content-type for the §8
  export.
- **Anchor cadence (v1.1).** MUST-anchor interval (per-N-links vs per-time)
  and a concrete minimum-notary conformance test.
- **Full JCS (v1.1).** Adopt RFC-8785 number/string canonicalization
  verbatim to widen the §2 domain beyond safe integers.
- **Key transparency.** Anti-equivocation proof for the §7 key set
  (CT-style) — RFC-007 candidate.
- **Cross-chain reciprocity.** Whether an RFC-003 envelope wrapping an
  RFC-006 ledger carries native links or the §5 projection.

## 12 · Compliance with companions

- **RFC-001 §9 → RFC-006 §3–4/§6.** Append-only + signed, strengthened to
  chain-linked + externally anchored.
- **RFC-004 → RFC-006 §5.** Every RFC-006 ledger is an RFC-004 log via the
  normative projection; no client breakage.
- **RFC-005 → RFC-006 §7.** Asymmetric attestations MUST use RFC-005
  encoding / endpoint / keyId.
