# RFC-006: Hash-chained ledger + external anchoring profile (extends RFC-004)

**Status:** Draft ¬∑ 2026-05. **Author:** Naza (naza@naza.ar).
**License:** CC-BY-4.0. **DOI:** pending Zenodo deposit.
**Companions:** RFC-001 (liability + governance taxonomy), RFC-004 (operational-log
spec ‚Äî the base this profile extends), RFC-005 (Ed25519 asymmetric upgrade).
**Reference implementation:** Vultur `@vultur/core/{audit,anchor}.ts`
(the live producer). **Independent verifier:** `tools/arg-verify/arg-verify.mjs`
(`chain`, `project`, `bundle`, `vectors` ‚Äî zero dependency, offline).

---

## 1 ¬∑ The gap RFC-006 fills

RFC-004 ¬ß4 specifies an append-only, per-entry HMAC-signed operational log and,
in ¬ß11, explicitly leaves open: *"Stores that allow random-access overwrite are
permitted only if a Merkle-chain or checksum tree gives the same guarantee"* and
flags aggregation / anti-equivocation as unresolved. A production sociedad-IA
that takes regulatory exposure seriously will want strictly more than per-entry
tamper-evidence:

- **Insertion / deletion / reordering detection**, not just per-record mutation
  detection. A per-entry HMAC proves *this record* was not edited; it does not
  prove the *set* of records is complete and ordered.
- **A guarantee against the operator itself.** Per-entry HMAC is computed with
  the operator's key; the operator can re-sign a rewritten history. The log
  must be defensible even when the adversary is the sociedad-IA.

RFC-006 specifies the profile that delivers both: a **linked HMAC hash-chain**
plus an **external anchor sub-chain**. It is a strict superset of RFC-004:
**every RFC-006 ledger projects, by a normative deterministic map (¬ß5), onto
RFC-004-conformant entries**, so a regulator holding only RFC-004 and the
published RFC-004 tooling can verify an RFC-006 producer without new software.

This document is what a legislator cites when they want the *stronger* floor,
and what a library author implements to claim RFC-006 conformance.

## 2 ¬∑ Canonical-JSON (normative, inherited)

RFC-006 uses the RFC-004 ¬ß3 canonical-JSON function verbatim: keys sorted
lexicographically at every level, arrays positional, primitives via
`JSON.stringify`. Equivalently, `JSON.stringify` of a recursively
key-sorted object for any JSON-safe value. The reference producer's
`canonicalize()` (`JSON.stringify(sort(v))`) is a conformant implementation;
`arg-verify` reimplements the ¬ß3 form clean-room and the two agree on every
published vector.

**Domain (normative).** Canonical-JSON is defined **only** over the JSON
data model: objects with string keys, arrays, strings, **finite** numbers,
booleans, and `null`. `undefined`, functions, symbols, `BigInt`, and
non-finite numbers (`NaN`, `¬±Infinity`) are **out of domain**. A conformant
producer MUST NOT present an out-of-domain value for canonicalization (audit
payloads already normalise absent `meta` to `null`); a conformant verifier
MUST **reject** such input rather than canonicalize it. Rationale: the
canonical string is the signed material, so any input on which two
conformant implementations could disagree is a cross-implementation
signature-ambiguity hole ‚Äî e.g. `JSON.stringify(sort(v))` drops an
`undefined` object member while a na√Øve hand-rolled serializer might emit a
literal `undefined` (invalid JSON, different bytes, different signature).
Restricting the domain closes the hole by construction: across the entire
JSON data model the producer's `JSON.stringify(sort(v))` and `arg-verify`'s
clean-room `canonical()` are byte-identical (asserted by every vector); they
are only permitted to differ outside it, which is now forbidden input.

**Key ordering (normative).** Object members are ordered by ascending
lexicographic comparison of the **key string's UTF-16 code units**, applied
at every nesting level ‚Äî *including* keys that look like integers. A
conformant serializer MUST build the canonical form from the sorted key
list directly and MUST NOT route the object through a runtime that reorders
keys, because ECMAScript `[[OwnPropertyKeys]]` enumerates array-index keys
(`"0"`, `"2"`, `"10"`, ‚Ä¶) in ascending *numeric* order before other keys ‚Äî
so `JSON.stringify(sort(v))` emits `2,10` where this spec requires `10,2`.
`arg-verify`'s string-built `canonical()` is the conformant reference;
`vectors` pins the integer-like-key case. (Producer-side caveat tracked in
CONFORMANCE.md.)

**Strings (normative).** Serialized exactly as ECMAScript `JSON.stringify`
(ECMA-262 `QuoteJSONString`: `"`, `\`, and U+0000‚ÄìU+001F escaped; all other
code points, including non-ASCII, emitted literally as UTF-8). **No Unicode
normalization is applied** ‚Äî byte content is significant, so the composed
(NFC) and decomposed (NFD) forms of the same grapheme are distinct signed
inputs (the reference producer and verifier agree on this; it is an
input-stability property, not an implementation divergence). A producer that
needs normalization-stable signatures MUST normalize (e.g. NFC) *before*
signing. This matches RFC 8785 (JCS), which likewise does not normalize.

**Numbers (normative).** The domain is finite numbers only; each is
serialized by the ECMAScript Number-to-String algorithm as used by
`JSON.stringify` (shortest round-tripping form; e.g. `1e+21`, `-0` ‚Üí `0`,
`1.0` ‚Üí `1`). A conformant non-ECMAScript implementation MUST reproduce that
exact formatting. The reference producer and `arg-verify` agree on every
number by construction (both delegate to `JSON.stringify`).

## 3 ¬∑ Chain link (normative)

The unit of an RFC-006 ledger is a **link**, not a free-standing entry. Each
link has this payload:

```
interface ChainLinkPayload {
  seq:        number;            // 1-based, contiguous, no gaps
  prevHash:   string;            // hash of link seq-1; "GENESIS" for seq 1
  societyId:  string | null;     // tenant; null = global ledger
  actor:      string;            // who/what initiated (tool, user, system)
  action:     string;            // the operation that produced the side effect
  meta:       unknown;           // operation detail; null if none
  ts:         string;            // ISO-8601 UTC, server clock at write time
}
```

The link's authenticator is:

```
hash_n = HMAC_SHA256( AUDIT_SECRET,
           canonical({ seq, prevHash, societyId, actor, action,
                       meta: meta ?? null, ts }) )            // lowercase hex
prevHash_1 = "GENESIS"
prevHash_n = hash_{n-1}        (n > 1)
```

`hash_n` is **raw lowercase hex** (no `sha256:` prefix; that prefix is RFC-004's
per-entry convention, not the chain convention). `meta` is normalized to `null`
when absent *before* canonicalization so sign and verify agree.

Stored link = payload + `hash`. The only permitted mutation primitive is
`append`. There is no `update`, no `delete` (TTL-purge for RFC-004 ¬ß7 retention
is the sole exception, and it purges whole prefixes, never interior links).

## 4 ¬∑ Chain verification (normative)

Two verification modes:

**4.1 Contiguous chain (full integrity).** For an ordered slice starting at the
genesis or at a known checkpoint, a verifier asserts, for every `i`:

1. `seq_i == seq_{i-1} + 1` (contiguity ‚Äî detects deletion / reordering).
2. `prevHash_i == hash_{i-1}` (linkage ‚Äî detects insertion / deletion).
3. `hash_i == HMAC(secret, canonical(payload_i))` (authenticity ‚Äî detects
   mutation, including deeply-nested mutation).

Any failure returns `{ valid:false, brokenAtSeq, reason }`. A passing slice
proves the records are unmutated **and** complete **and** ordered.

**4.2 Per-record (non-contiguous slice).** For a filtered view (e.g. one
society's links pulled out of a global chain), contiguity cannot hold; the
verifier asserts only check (3) per record. This proves each record is
unmutated but not set-completeness ‚Äî the verifier MUST label such a result
`recordsOnly: true`.

## 5 ¬∑ RFC-004 projection (normative ‚Äî this is the conformance bridge)

An RFC-006 producer is RFC-004-conformant **by construction** via this
deterministic projection `P`. Given chain link `L`, define RFC-004
`OperationalLogEntry` `P(L)`:

```
id         := `${L.ts}-${L.hash.slice(0,8)}`
sessionId  := if   typeof L.societyId === "string"
                   && /^[A-Za-z0-9_-]{8,64}$/.test(L.societyId)
              then L.societyId
              else if L.societyId == null  then "GLOBAL-LEDGER"
              else "soc-" + base64url(sha256(String(L.societyId))).slice(0,16)
ts         := L.ts
tool       := L.action
governance := (L.meta is object
               && L.meta.governance ‚àà RFC-004 ¬ß6 enum)
              ? L.meta.governance : "audit-logged"
input      := { actor: L.actor, seq: L.seq, meta: L.meta ?? null }
output     := (omitted)
hmac       := "sha256:" + HMAC_SHA256( PROJECTION_SECRET,
                            canonical(P(L) without `hmac`) )
```

Properties (all machine-checked by `arg-verify project` / `vectors`):

- **Determinism.** `P` is a pure function of `L`. Same chain ‚Üí same entries.
- **RFC-004 validity.** `P(L)` is a well-formed `OperationalLogEntry`;
  `verifyEntry(P(L), PROJECTION_SECRET)` per RFC-004 ¬ß3 returns `true`.
- **Injectivity.** `input.seq` + `id` (which embeds `hash`) make `P`
  collision-free, so the projected log preserves chain order and identity.
- **Lossy-but-declared.** `governance` defaults to `audit-logged` when the
  producer did not carry an explicit class in `meta.governance`. An RFC-006
  producer that wants lossless projection MUST set `meta.governance` to an
  RFC-004 ¬ß6 value. This is the *only* lossy point and it fails safe (toward
  the more-liability class, never toward `mocked-upstream`).

`PROJECTION_SECRET` MAY equal `AUDIT_SECRET` or be a separate RFC-004 signing
key; the projection is a *view*, the native chain (¬ß3‚Äì4) remains the stronger
guarantee. A regulator running RFC-004 tooling
(`arg-verify entry P(L).json --secret PROJECTION_SECRET`) gets a green check
with zero RFC-006 awareness.

## 6 ¬∑ External anchoring (normative)

Periodically the producer checkpoints the chain head into an **anchor**, and
anchors form their own HMAC-signed chain:

```
interface AnchorBody {
  seq:        number;      // 1-based, contiguous
  headSeq:    number;      // chain head seq at checkpoint time
  headHash:   string;      // chain head hash at checkpoint time
  prevAnchor: string;      // previous anchor signature; "GENESIS" for seq 1
  ts:         string;      // ISO-8601 UTC
}
signature_n = HMAC_SHA256( AUDIT_SECRET, canonical(AnchorBody_n) )   // hex
prevAnchor_n = signature_{n-1}
```

Anchor verification mirrors ¬ß4.1 (contiguity + `prevAnchor` linkage +
signature recompute). The anchor chain SHOULD be mirrored to an **external
notary** outside the operator's control (an append-only third party,
timestamping authority, or public log). Effect: the operator cannot
retroactively rewrite or backdate history without invalidating every anchor
issued since the divergence point, and the external mirror is evidence the
operator cannot suppress. This is the ¬ß1 "guarantee against the operator
itself."

### 6.1 ¬∑ OpenTimestamps public-anchor profile (normative, additive)

¬ß6's anchor signature is HMAC-SHA256 under the operator's own `AUDIT_SECRET`.
That alone is **not** proof against the operator: a holder of the secret can
recompute a fully self-consistent fake history and fake anchor chain. ¬ß6's
tamper-evidence is therefore "to witnesses who retained an anchor", not to a
fresh verifier. ¬ß6.1 closes that gap **without depending on witnesses** by
committing each anchor to a public, operator-independent timestamp.

A conformant producer MAY publish, for any anchor, an **OpenTimestamps (OTS)
proof** that commits the **anchor digest** to the public Bitcoin calendars:

```
digest_n = SHA256( canonical(AnchorBody_n) )      // hex, 32 bytes
```

This is the **same canonical material** ¬ß6 HMAC-signs, so the OTS proof and the
HMAC anchor commit to one object (the anchor record, including its `prevAnchor`
linkage). The producer submits `digest_n` to one or more public OTS calendars
and stores the returned `.ots` proof. A proof begins **`pending`** (the
calendar received the commitment) and **upgrades to `bitcoin`** once a block
confirms it (carrying a block height + Merkle path to a Bitcoin block header),
over minutes-to-hours, with no further producer action.

Effect: the operator cannot backdate `digest_n` past the timestamp of the
Bitcoin block that confirms it, and cannot forge that block. The trust root is
Bitcoin's proof-of-work, **not any ar-agents key** ‚Äî the proof is verifiable by
anyone, with no trust in the producer and no retained witness.

**Delivery.** The producer SHOULD expose the raw proof bytes at a stable URL so
a third party can verify with the standard tooling and zero producer code:

```
GET /api/audit/anchor/{seq}/ots        -> application/octet-stream (.ots file)
GET /api/audit/anchor?upgrade=1        -> re-query calendars, upgrade pending proofs
GET /api/audit/anchor                  -> includes `proofs` keyed by anchor.seq
```

**Verification.** The canonical full check is the official `ots verify` CLI
(commitment -> Bitcoin block header). For an offline commitment check without
the CLI, `arg-verify timestamp <anchor.ots> --digest <digestHex>` confirms the
`.ots` commits to the supplied digest and reports the embedded attestation
(`pending` / `bitcoin` + block height). This command is **OPTIONAL** and is
**excluded from `arg-verify vectors`**: it reads a binary `.ots` file, so
including it would break the zero-dependency, offline `vectors` invariant. The
trust-free **floor** remains the ¬ß7 Ed25519 attestation; OTS is the
public-anchor **upgrade**.

**Attestation binding (additive).** A ¬ß7 attestation MAY carry an optional
`body.timestamp` referencing the OTS-stamped anchor that covers its head:

```
timestamp?: {
  type: "opentimestamps",
  anchorSeq: number,           // the OTS-stamped anchor (headSeq >= attested head)
  digest:    string,           // SHA256(canonical(AnchorBody)) of that anchor
  status:    "pending" | "bitcoin",
  bitcoinBlockHeight?: number
}
```

Because canonical-JSON (¬ß2) sorts keys, adding `timestamp` changes the signed
bytes only for **newly-issued** attestations; historical signatures and the
published conformance vectors (whose body has no `timestamp`) are unaffected. A
verifier MUST read `body.timestamp` **defensively** (it may be absent) and MUST
NOT require it for the trust-free check.

## 7 ¬∑ Asymmetric attestation (normative, aligns RFC-005)

An RFC-006 producer MAY additionally publish a signed compliance attestation
over the chain head (the reference producer's `vultur.compliance.attestation`).
When it does, RFC-006 **requires RFC-005 wire conventions** so a single
verifier works everywhere:

- Ed25519 signature value: **base64url, unpadded** (not standard base64).
- Public key set published at **`/.well-known/sociedad-ia/keys`** as the
  RFC-005 ¬ß4 JSON (`keyId`, `alg`, `publicKey` SPKI base64url, `validFrom`,
  `validUntil`), in addition to any operator-specific endpoint.
- Each signature carries a **`keyId`** resolvable in that set, enabling
  rotation without invalidating historical attestations.

(The reference producer currently emits base64 + an embedded key at
`/api/audit/pubkey`. RFC-006 ¬ß7 is the normative target; the divergence is
tracked in CONFORMANCE.md and is a pure encoding/endpoint change.)

## 8 ¬∑ Verification interface (extends RFC-004 ¬ß5)

An RFC-006 producer exposes, in addition to RFC-004 ¬ß5:

- `GET /api/audit/verify` ‚Üí contiguous-chain verification result
  (`{ valid, count, brokenAtSeq?, reason? }`).
- `GET /api/audit/anchor` (or equivalent) ‚Üí the anchor chain for ¬ß6
  verification.
- `GET /api/audit/{slug}/attestation` ‚Üí the ¬ß7 attestation.
- A **projection export** that emits `P(L)` for a session/slug so a regulator
  can run RFC-004 tooling unchanged. RFC-006 v1.1 will normative-ize the exact
  route; v1 requires only that such an export exists and is documented.

### 8.1 ¬∑ Export bundle (normative ‚Äî the regulator artifact)

The artifact a data subject / regulator actually downloads (the reference
producer's Ley-25.326 *derecho de acceso* export) is a **bundle**:

```
{ exportedAt, society, movements, invoices,
  auditEvents,                 // this society's links, ordered by seq asc
  ledgerVerification,          // producer's recordsOnly result for the slice
  attestation,                 // ¬ß7 SignedAttestation over the chain head
  notice }
```

Normative mapping so an independent verifier can check the *bundle as
received*, not a hand-extracted fragment:

1. **Timestamp column.** Export records carry **`createdAt`**, not `ts`. The
   ¬ß3 canonical hash material's `ts` is `createdAt` rendered ISO-8601 UTC
   (`new Date(createdAt).toISOString()`). A verifier MUST accept `ts` when
   present and otherwise derive it from `createdAt`; the two forms hash
   identically.
2. **recordsOnly slice.** `auditEvents` is a per-society filter of the global
   chain and is therefore **non-contiguous**; it MUST be verified under ¬ß4.2
   (per-record), and the result labelled `recordsOnly`. Contiguity/linkage
   MUST NOT be asserted over a per-society export.
3. **Attestation‚Üîbundle binding (MUST).** The ¬ß7 attestation signs only the
   head summary, so a valid attestation can be lifted onto a swapped bundle.
   A conformant verifier MUST reject unless, in addition to the Ed25519
   check: `attestation.body.chain.societyEventCount == auditEvents.length`,
   the attestation society identity equals the bundle society
   (`slug`, and `cuit` when present), and
   `attestation.body.chain.verification` canonical-equals
   `ledgerVerification`.
4. **Trust-free floor.** The Ed25519 attestation + binding (1,3) require **no
   secret** and are the third-party guarantee. The recordsOnly HMAC (2) is
   operator-keyed; a verifier without the secret MUST skip it explicitly
   rather than fail, and report that the trust-free guarantee still holds.

Reference command: `arg-verify bundle <vultur-export-SLUG.json>
[--secret S] [pubkeyB64]`.

## 9 ¬∑ Conformance test vectors

`/test-vectors/rfc-006-v1.json` (published) contains, with byte-exact expected
outputs computed deterministically:

1. **Chain vectors.** A genesis-rooted 3-link chain with expected `hash` per
   link; `verifyChain` MUST return `valid:true`. A mutated copy
   (`meta` changed) MUST return `valid:false` at the mutated `seq`. A
   deletion copy (middle link removed) MUST return `valid:false` with
   `reason` indicating linkage/contiguity break.
2. **Anchor vectors.** A 2-anchor chain over the ¬ß9.1 head with expected
   `signature` per anchor; tamper MUST break verification.
3. **Projection vectors.** For each ¬ß9.1 link, the exact `P(L)` object and its
   RFC-004 `hmac`. A library claims RFC-006 conformance iff: it reproduces
   every `hash`, every anchor `signature`, every `P(L)` object
   (canonical-equal), and every projected `hmac` (string-equal), AND the
   projected entries pass RFC-004 ¬ß3 `verifyEntry`.
4. **Export-bundle vectors (¬ß8.1).** A real-shaped `vultur-export-SLUG`
   bundle (non-contiguous society slice keyed by `createdAt`, ¬ß7 attestation
   nested under `.attestation`, fixed Ed25519 keypair) plus a tampered twin.
   A conformant verifier MUST: Ed25519-verify the embedded attestation;
   enforce the ¬ß8.1(3) binding; recordsOnly-verify the slice with the
   `createdAt‚Üíts` mapping; and, on the tampered twin, keep the attestation
   valid yet flag the mutated record (demonstrating why binding is MUST).

Run: `node tools/arg-verify/arg-verify.mjs vectors` ‚Äî expected: all PASS.
A regulator with only the published bundle runs
`node tools/arg-verify/arg-verify.mjs bundle vultur-export-SLUG.json`.

## 10 ¬∑ Decision request

For the AR sociedad-IA legislative project:

- RFC-004 v1 remains the **minimum** operational-log floor.
- RFC-006 v1 is the **recommended profile** for any sociedad-IA handling real
  value, regulated activity, or third-party funds. An RFC-006 producer is
  RFC-004-conformant by the ¬ß5 projection, so citing RFC-006 does **not**
  fork the standard ‚Äî it raises the floor while staying RFC-004-checkable.
- Suggested legislative language: *"the operational log MUST satisfy RFC-004
  v1; producers handling [regulated thresholds] MUST satisfy RFC-006 v1."*

## 11 ¬∑ Open questions

- **Projection route (v1.1).** Normative path + content-type for the ¬ß8
  projection export.
- **Anchor cadence (v1.1).** MUST-anchor interval (per-N-links vs per-time)
  and minimum external-notary properties (append-only proof, independence).
- **`meta.governance` enforcement.** Should RFC-006 v1.1 make
  `meta.governance` MUST (lossless projection) rather than SHOULD?
- **Key transparency.** RFC-005 ¬ß8 carries over: anti-equivocation proof for
  the ¬ß7 key set (Certificate-Transparency-style) is an RFC-007 candidate.
- **Cross-chain reciprocity.** RFC-003 envelope wrapping an RFC-006 ledger:
  define whether the envelope carries native links or the ¬ß5 projection.
